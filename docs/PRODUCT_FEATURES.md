# 产品功能完善建议

> 本文档从产品功能角度分析当前系统需要完善的功能模块，按优先级排序。

**当前状态**: Phase 3 已完成（测试覆盖率 85%+，282 个测试用例通过）

**最后更新**: 2026-03-03

---

## 一、核心交易功能增强

### 1.1 风控系统 🔴 缺失

| 功能点 | 描述 | 优先级 |
|--------|------|--------|
| 单日交易限额 | 防止程序错误导致的大额交易损失 | P0 |
| 仓位限制 | 单只基金最大持仓比例控制 | P0 |
| 黑名单机制 | 禁止交易的基金列表 | P0 |
| 异常检测 | 检测异常交易行为并自动暂停策略 | P0 |
| 累计亏损限额 | 日/周/月最大亏损阈值 | P1 |

**实现建议**:
- 在 `packages/backend/src/core/risk/` 目录下创建风控模块
- 新增 `RiskControlService` 提供统一的風控检查接口
- 在交易执行前增加 `checkRiskLimits()` 调用
- 新增 `risk_limits` 配置表存储用户自定义限额

---

### 1.2 交易确认机制 🔴 缺失

| 功能点 | 描述 | 优先级 |
|--------|------|--------|
| 大额交易人工确认 | 超过阈值的交易需要二次确认（Telegram/飞书回复确认） | P0 |
| 交易前检查 | 余额不足、持仓不足时提前预警 | P0 |
| 交易失败重试 | 失败订单的自动重试逻辑（指数退避） | P1 |
| 交易确认超时 | 超时未确认自动取消交易 | P2 |

**实现建议**:
- 在 `packages/backend/src/core/trading/` 目录下创建交易确认模块
- 新增 `TradingConfirmationService` 处理确认逻辑
- 利用现有的 Bull 队列实现延迟确认任务
- 通知系统增加"确认按钮"交互支持

---

### 1.3 订单管理 🟡 部分缺失

| 功能点 | 描述 | 优先级 |
|--------|------|--------|
| 订单取消功能 | 在平台未确认前取消订单 | P1 |
| 订单实时追踪 | 实时轮询/ webhook 更新订单状态 | P1 |
| 撤单记录 | 记录撤单历史到 `Transaction` 表 | P2 |
| 订单批量操作 | 批量取消/查询订单 | P2 |

**实现建议**:
- 扩展 `Transaction` 实体增加 `cancelled_at`、`cancel_reason` 字段
- 在 `TiantianBrokerService` 增加 `cancelOrder()` 方法
- 新增定时任务轮询待确认订单状态

---

## 二、数据分析与可视化

### 2.1 收益分析 🔴 缺失

| 功能点 | 描述 | 优先级 |
|--------|------|--------|
| 收益曲线图 | 累计收益随时间变化曲线 | P1 |
| 收益归因分析 | 分析各策略/基金对总收益的贡献 | P1 |
| 月度/年度收益统计 | 按时间维度统计收益率 | P1 |
| 收益率对比 | 对比基准指数（沪深 300、中证 500） | P2 |

**实现建议**:
- 后端新增 `AnalyticsService` 提供收益计算接口
- 前端使用 `recharts` 或 `chart.js` 绘制图表
- 新增 `daily_portfolio_value` 表记录每日资产快照

---

### 2.2 持仓分析 🔴 缺失

| 功能点 | 描述 | 优先级 |
|--------|------|--------|
| 持仓分布图 | 基金类型、行业、风险等级分布 | P1 |
| 持仓成本分析 | 分批买入成本对比图 | P1 |
| 盈亏分析 | 已实现/未实现盈亏详细拆解 | P1 |
| 持仓占比变化 | 历史持仓权重变化趋势 | P2 |

**实现建议**:
- 前端使用饼图、柱状图展示分布
- 后端提供 `/api/analytics/positions` 聚合接口
- 复用现有 `Position` 实体，增加历史快照

---

### 2.3 交易记录分析 🔴 缺失

| 功能点 | 描述 | 优先级 |
|--------|------|--------|
| 交易统计 | 买入/卖出次数、成功率、平均收益率 | P2 |
| 手续费统计 | 累计交易成本分析 | P2 |
| 交易日志 | 详细的操作日志（时间、金额、净值、手续费） | P2 |
| 交易日历 | 可视化展示交易日期分布 | P3 |

**实现建议**:
- 后端提供 `/api/analytics/transactions` 统计接口
- 前端使用表格 + 图表组合展示
- 利用现有 `Transaction` 实体聚合查询

## 三、策略系统增强

### 3.1 策略回测可视化 🟡 前端已实现基础功能

| 功能点 | 描述 | 优先级 |
|--------|------|--------|
| 收益对比图 | 策略收益 vs 基准指数 | P2 |
| 回撤曲线图 | 历史回撤可视化 | P2 |

### 3.2 策略组合 🔴 缺失

| 功能点 | 描述 | 优先级 |
|--------|------|--------|

**实现建议**:

- 策略执行时检查资金占用情况
- 资金分配服务统一调度

---

### 3.3 智能策略 🔴 缺失

| 功能点 | 描述 | 优先级 |
|--------|------|--------|
| 均线策略 | 基于 MA/EMA 的买卖信号 | P2 |
| 估值策略 | 基于 PE/PB 的低估买入、高估卖出 | P2 |
| 动量策略 | 追涨杀跌逻辑（N 日涨幅排名） | P3 |
| 网格策略增强 | 自适应网格、动态调整 | P2 |

**实现建议**:

- 外部数据接入（估值数据需要额外的 API）
- 策略工厂模式统一管理

## 五、系统运维

### 5.1 监控告警 🔴 缺失

| 功能点 | 描述 | 优先级 |
|--------|------|--------|
| 系统健康检查 | 数据库、Redis、浏览器会话状态 | P1 |
| 错误告警 | 关键错误实时通知 | P1 |
| 性能监控 | API 响应时间、任务执行时间 | P2 |
| 资源监控 | CPU、内存、磁盘使用率 | P2 |

**实现建议**:
- 新增 `HealthController` 提供 `/health` 端点
- 使用 `@nestjs/terminus` 健康检查库
- 关键服务增加 Prometheus 指标

---

### 5.2 日志系统 🟡 NestJS 默认日志

| 功能点 | 描述 | 优先级 |
|--------|------|--------|
| 操作日志 | 记录所有用户操作（谁、何时、做了什么） | P1 |
| 交易日志 | 详细交易过程记录（请求/响应） | P1 |
| 日志查询 | 日志搜索、过滤、导出 | P2 |
| 结构化日志 | JSON 格式便于 ELK 收集 | P2 |

**实现建议**:
- 使用 `winston` 或 `pino` 替代默认日志
- 新增 `OperationLog` 实体存储关键操作
- 日志中间件记录请求/响应

## 优先级总结

| 优先级 | 模块数量 | 预计工时 | 说明 |
|--------|---------|---------|------|
| **P0** | 2 | 3-4 天 | 风控系统、交易确认（安全相关） |
| **P1** | 8 | 7-10 天 | 核心功能增强、数据分析基础 |
| **P2** | 12 | 10-14 天 | 体验优化、运维工具 |
| **P3** | 5 | 5-7 天 | 增值功能、高级特性 |

**总计**: 约 25-35 天完整工期
